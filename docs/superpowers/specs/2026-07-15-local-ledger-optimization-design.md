# 本地家庭账本可靠性与使用闭环设计

## 背景

当前项目已经覆盖家庭成员、资产账户、自定义账户类型、分类、收支流水、预算、投资、负债、资产快照，以及支付宝 CSV 和微信 xlsx 导入。下一阶段重点不是增加更多页面，而是降低重复导入和未整理流水造成的数据风险，并让仪表盘能够直接引导用户完成日常处理。

本项目暂时只在本机使用。本轮不实现登录、权限、多家庭隔离、云同步、备份或恢复功能。

## 目标

1. 相同账单文件不能被重复导入，已完成的导入可以整批撤销。
2. 导入前能确认文件、目标成员、目标账户、笔数与金额；导入后能看到批次结果。
3. 待分类、未关联账户和疑似重复流水集中到一个待处理入口，并支持批量处理。
4. 账户核账信息只展示可被现有数据模型可靠计算的内容，不伪造完整账面余额。
5. 仪表盘将抽象的风险提示替换为可点击、可处理的事项。
6. 设置页在数据增长后仍可搜索和理解，历史引用不因停用配置而丢失。
7. 页面按需请求数据，避免切换月份或修改一条记录时刷新所有模块。

## 非目标

- 不实现公网部署所需的认证和授权。
- 不实现备份、恢复、导出或云同步。
- 不把账本改造成复式记账系统。
- 不根据流水自动改写资产账户当前价值；账户价值仍由编辑和快照维护。
- 不删除或重写既有历史数据。

## 分阶段边界

### 阶段一：导入可靠性与批次撤销

扩展现有 `ImportJob`，保存文件名、SHA-256 文件指纹、来源类型、成员、目标账户、状态、汇总数量和金额。`FinanceTransaction` 增加可空的 `importJobId`，用于追溯导入批次。

前端读取文件后计算 SHA-256，并在确认导入前提交文件信息和解析后的流水。后端在单个数据库事务中完成以下操作：

1. 检查同一家庭是否已有状态为 `completed` 的相同文件指纹。
2. 创建状态为 `processing` 的导入批次。
3. 校验成员、账户、分类和每条流水。
4. 创建流水并关联批次。
5. 写入成功汇总，将批次更新为 `completed`。

任何一步失败都回滚整个事务。重复文件返回明确的冲突错误，并包含原导入批次的日期和笔数。

撤销批次时，在一个事务内软删除该批次仍有效的流水，并将批次状态更新为 `reverted`。已经撤销的批次不能再次撤销；撤销后允许用户重新导入相同文件。手工编辑过的导入流水仍随批次撤销，因为“撤销导入”语义是恢复到导入前状态，界面需要在确认框中明确说明这一点。

导入抽屉新增目标账户选择、文件汇总、收入/支出金额和重复文件提示。交易页新增“导入记录”入口，展示最近批次及撤销按钮。

### 阶段二：待处理与核账提示

待处理事项从现有数据派生，不新增容易失真的持久状态：

- 待分类：分类名称为 `待分类支出`，或分类引用为空且不是转账的有效流水。
- 未关联账户：`accountId` 为空的有效流水。
- 疑似重复：同一成员、日期、类型、金额和备注完全相同，且不属于同一导入批次的有效流水。系统只提示，不自动删除。

交易页增加“待处理”视图，按问题类型筛选，支持批量设置分类、成员或账户。批量更新由后端专用接口在事务中完成，返回实际更新数量；任何目标配置不存在时整批失败。

账户核账不假设转账方向。每个账户展示：

- 最近快照日期和距今天数。
- 最近快照后已关联的收入、支出与可计算净流量。
- 最近快照后转账数量；存在转账时明确显示“净流量未包含转账方向”。
- 未关联账户流水总数，作为全局待处理提示。
- 当前价值与最近快照价值的变化，仅作为人工核对参考。

仪表盘新增按优先级排序的可点击行动项：待分类笔数、未关联账户笔数、疑似重复组数、超预算分类数、超过 30 天未保存快照的账户数。点击后跳转到对应页面和筛选状态。

### 阶段三：设置页和加载性能

设置页保持“家庭成员 + 资产账户类型”与“分类”两列布局。账户类型和分类增加本地即时搜索、使用数量和稳定排序。操作文案从“删除”改为“停用”；停用前显示当前被多少账户、流水或预算引用。停用配置不影响历史记录，创建和编辑表单只提供启用配置。

账户历史快照查询不再依赖账户必须处于启用状态。已删除账户的快照显示原账户名称，并标记“已停用账户”。账户名称需要随快照查询从包括软删除记录的账户集合中解析。

前端数据加载拆分为以下边界：

- 全局配置：家庭成员、账户类型、分类，首次使用时加载并缓存。
- 仪表盘：汇总、趋势和行动项。
- 资产账户：账户、快照和核账摘要。
- 收支记录：流水、预算、待处理汇总和导入批次。
- 投资与负债：仅各自页面进入时加载。

写操作只失效受影响的数据域。例如编辑分类刷新全局配置和当前交易视图，但不重新请求投资和负债。继续使用现有 React 状态与 API 客户端，不新增状态管理依赖。

## 接口边界

新增或调整的接口遵循现有 `/api` 风格：

- `POST /transactions/imports`：导入账单并返回批次摘要。
- `GET /transactions/imports`：列出最近导入批次。
- `DELETE /transactions/imports/:id`：撤销导入批次。
- `GET /transactions/action-items`：返回待处理数量和疑似重复分组。
- `PATCH /transactions/batch`：批量更新分类、成员或账户。
- `GET /accounts/reconciliation`：返回账户快照新鲜度和快照后净流量。
- `GET /dashboard/action-items?month=YYYY-MM`：返回可跳转的仪表盘行动项。

原 `POST /transactions/import` 在前端迁移完成后保留兼容入口，内部调用新的批次导入逻辑，避免现有调用立即失效。

## 数据一致性与错误处理

- 导入、批量更新和撤销导入必须使用数据库事务。
- 金额继续使用共享 `MoneyAmount` 类型和整数分，禁止浮点累加。
- 文件指纹由前端计算、后端校验格式；防重依据文件内容而不是文件名。
- 停用配置和删除账户继续使用软删除，不级联删除历史流水或快照。
- 接口错误向前端返回可直接展示的中文业务信息；未知错误保留通用提示，不暴露数据库细节。
- 空状态分别说明“没有数据”和“没有待处理事项”，不得用同一个文案。

## 测试策略

后端使用 Vitest 覆盖：重复文件阻止导入、撤销批次、撤销后重新导入、事务失败不留部分流水、批量更新全有或全无、已删除账户快照仍可查询、核账摘要排除转账方向。

前端使用 Vitest 覆盖：文件 SHA-256 和导入预览汇总、待处理筛选、行动项跳转参数、设置搜索与使用数量、按路由选择加载器和缓存失效。

最终验证包括完整测试、类型检查、生产构建，以及浏览器中的以下流程：

1. 导入文件并查看批次结果。
2. 重复导入同一文件并看到阻止提示。
3. 撤销批次后确认对应流水消失。
4. 在待处理视图批量分类和关联账户。
5. 从仪表盘行动项跳转到正确筛选。
6. 搜索和停用分类或账户类型。
7. 桌面与移动宽度下检查无重叠、裁切或不可操作控件。

## 验收标准

- 相同文件在未撤销前无法重复导入。
- 撤销导入不会影响手工创建或其他批次的流水。
- 待处理数量与对应筛选后的有效流水一致。
- 含转账的账户不会显示虚假的精确核账差额。
- 停用的分类和账户类型仍能在历史记录中正确显示。
- 已删除账户的历史快照仍可查询，并带停用标记。
- 进入任一页面时不再请求无关业务模块的数据。
- 所有新增后端写操作均有事务回滚测试。

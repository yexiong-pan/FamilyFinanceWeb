# 收支筛选条件路由化设计

## 目标

将支出和收入明细页的全部筛选条件写入 URL，使页面刷新、链接复制和浏览器前进后退都能恢复相同筛选状态。切换月份时保留筛选条件。

## URL 契约

支出和收入页面共用以下查询参数：

| 参数 | 含义 | 有效值 |
| --- | --- | --- |
| `month` | 当前月份 | `YYYY-MM` |
| `category` | 分类名称 | 非空字符串 |
| `member` | 家庭成员名称 | 非空字符串 |
| `status` | 确认状态 | `pending`、`confirmed` |
| `min` | 最低金额 | 大于等于 0 的有限数字 |
| `max` | 最高金额 | 大于等于 0 的有限数字 |

示例：

```text
/spending/details?month=2026-07&category=餐饮&member=雄哥&status=confirmed&min=10&max=500
/income/details?month=2026-07&category=工资薪酬&status=confirmed
```

空筛选不写入 URL。无效状态、负数金额和非数字金额在解析时忽略，并在规范化 URL 时移除。分类和成员名称使用标准 URL 编码。

## 状态和导航

URL 是收支筛选条件的唯一数据源。`CashflowPage` 接收已解析的筛选对象和筛选更新回调，不再维护分类、成员、状态和金额范围的本地副本。

- 筛选控件变化时更新当前 URL，并使用 `history.replaceState`，避免连续操作产生大量历史记录。
- 页面、子 Tab 和月份切换继续使用 `history.pushState`。
- 在同一个支出或收入页面切换月份时，保留当前筛选参数。
- 在分类汇总和明细之间切换时保留筛选参数；汇总页不显示筛选控件，也不应用明细筛选。
- 从分类汇总点击分类时，跳转到明细并将对应分类写入 URL。
- 从支出切换到收入或从收入切换到其他主页面时，不携带当前收支筛选，避免不同业务页面互相污染。
- `popstate` 发生时同时恢复路由、月份和筛选条件。

## 代码边界

在独立的纯函数模块中实现：

1. 从 `URLSearchParams` 解析收支筛选。
2. 将筛选写入 `URLSearchParams`，同时保留 `month` 等页面级参数。
3. 生成规范化查询字符串，供首次加载和 `popstate` 使用。

`AppShell` 负责浏览器历史和 URL 更新；`CashflowPage` 只消费筛选值并触发更新，不直接操作 `window.history`。

- 清空筛选时删除对应参数。
- 切换月份和收支子 Tab 时保留筛选。
- 分类汇总点击分类生成明细路由及分类参数。
